Templates (twig)¶

Son ficheros que sirven para generar ficheros de texto en distintos formatos (HTML, XML, YML, JSON) y cuya contenido es calculado dinámicamente.

Ejemplo de plantilla PHP:

<!DOCTYPE html>
<html>
    <head>
        <title>Welcome to Symfony!</title>
    </head>
    <body>
        <h1><?php echo $page_title ?></h1>

        <ul id="navigation">
            <?php foreach ($navigation as $item): ?>
                <li>
                    <a href="<?php echo $item->getHref() ?>">
                        <?php echo $item->getCaption() ?>
                    </a>
                </li>
            <?php endforeach; ?>
        </ul>
    </body>
</html>

El problema de las plantillas PHP¶

  • Al ser código PHP permiten incluir código ejecutable que no pertenece a la vista (mala práctica).
  • No son las más legibles, son bastante “verbosas”.

Solución: Twig (http://twig.sensiolabs.org/)

Ejemplo de la plantilla anterior en twig:

<!DOCTYPE html>
<html>
    <head>
        <title>Welcome to Symfony!</title>
    </head>
    <body>
        <h1>{{ page_title }}</h1>

        <ul id="navigation">
            {% for item in navigation %}
                <li><a href="{{ item.href }}">{{ item.caption }}</a></li>
            {% endfor %}
        </ul>
    </body>
</html>

Síntaxis básica de twig¶

{{ parametro }} -> pinta el valor de la variable parametro.

{% comando %} ... {% endcomando %} -> ejecuta la acción expresada por comando en el bloque definido desde su declaración hasta {% endcomando%}.

{# this is a comment #} -> comentario.

Twig soporta filtros que modifican de alguna manera el contenido de una variable. (http://twig.sensiolabs.org/doc/filters/index.html);

{{ title|upper }}

Y funciones :

{{ cycle(['odd', 'even'], i) }}

Funcionamiento de twig (caching)¶

Las plantilla twig se compilan en código PHP. Dicho código es el que realmente utiliza la aplicación para construir la salida (HTML, XML, etc).

Por tanto lo primero que hace twig, cuando se tiene que usar una plantilla, es compilarla. Este proceso de compilación se hace una sola vez y se utiliza la versión compilada en PHP cada vez que haya que usar la plantilla. Los ficheros compilados se guardan en el directorio app/cache.

Por eso si se realiza un cambio en alguna plantilla twig, hay que borrar la cache para que se vuelva a recompilar. Afortunadamente, en el entorno de desarrollo, cada vez que se hace un cambio a una plantilla twig, se vuelve a recompilar automáticamente. Esto supone una pérdida de rendimiento, pero en un entorno de desarrollo está justificado y merece la pena.

(http://twig.sensiolabs.org/doc/internals.html)

Herencia¶

El sistema de plantillas twig, está provisto de un mecanismo de herencia gracias al cual la decoración de plantillas resulta de una flexibilidad y versatilidad total.

En twig la herencia se implementa mediante el concepto de bloque. En las plantillas podemos delimitar bloques que comienzan con un {% block nombre_bloque %} y finalizan con {% endblock %}. Las plantillas heredan todas las funcionalidades de las plantillas que extienden y pueden cambiar el código de los bloques heredadados.

Si una plantilla hija no define el bloque de una plantilla padre, se utiliza el contenido definido en la plantilla padre. En caso de que si defina tal bloque, se utiliza el contenido de la hija. Y si, en lugar de sustituir, queremos añadir más cosas al bloque de la plantilla padre, usamos {{ parent() }} en el bloque de la plantilla hija.

La herencia de plantillas puede llevarse a cabo a varios niveles, esto es, una plantilla puede heredar de otra plantilla que a su vez hereda de otra plantilla, etcétera. No obstante no se recomienda llevar a cabo muchos niveles de herencia, ya que puede llegar a ser bastante confuso e incontrolable. La estrategia que recomiendan los creadores de Symfony2 es usar tres niveles de herencia:

  • en el primer nivel se colocan la estructura básica del documento HTML, se corresponde con lo que hace la plantilla ::base.html.twig,
  • en el segundo se colocan los elementos específicos de cada sección del sitio, por ejemplo el menú de la sección,
  • en el tercero se reserva para los elementos propios de la acción, se corresponde con nuestra plantilla JazzywebAulasMentorAlimentosBundle:Default:index.html.twig

(Ver código en las aplicaciones del curso)

Nombres y ubicaciones de las plantillas¶

Las plantillas pueden ubicarse en dos sitios:

  • En app/Resources/views/. Aquí se colocan las plantillas globales, es decir, que pueden ser utilizadas por cualquier bundle de la aplicación.
  • En path/to/bundle/Resources/views/. Aquí van las propias de cada bundle.

La sintaxis para referirse a las plantillas es:

bundle:controller:template, por ejemplo: JAMNotasFrontendBundle:Notas:layout-etiquetas-notas.html.twig.

Además cada plantilla debe indicar son sufijos el formato y el motor de plantillas que usa.

Etiquetas (tags) y helpers¶

Inclusión de otras plantillas¶

{# src/Acme/ArticleBundle/Resources/views/Article/articleDetails.html.twig #}
<h2>{{ article.title }}</h2>
<h3 class="byline">by {{ article.authorName }}</h3>

<p>
    {{ article.body }}
</p>

La anterior plantilla puede ser incluida usando la etiqueta include:

{# src/Acme/ArticleBundle/Resources/views/Article/list.html.twig #}
{% extends 'AcmeArticleBundle::layout.html.twig' %}

{% block body %}
    <h1>Recent Articles<h1>

    {% for article in articles %}
        {{ include('AcmeArticleBundle:Article:articleDetails.html.twig', {'article': article}) }}
    {% endfor %}
{% endblock %}

A la plantilla incluida se le pasan los parámetros que requiere para pintarse. Para pasar dichos parámetros se utiliza sintaxis JSON.

Inclusión de controladores (plantillas con lógica de negocio)¶

{# app/Resources/views/base.html.twig #}

{# ... #}
<div id="sidebar">
    {{ render(controller('AcmeArticleBundle:Article:recentArticles', { 'max': 3 })) }}
</div>

Definicendo enlaces¶

{# src/Acme/ArticleBundle/Resources/views/Article/recentList.html.twig #}
{% for article in articles %}
    <a href="{{ path('article_show', {'slug': article.slug}) }}">
        {{ article.title }}
    </a>
{% endfor %}

article_show es el nombre de una ruta.

Se puede generar rutas absolutas usando url en lugar de path.

Definicendo enlaces a activos (assets)¶

<img src="{{ asset('images/logo.png') }}" alt="Symfony!" />

<link href="{{ asset('css/blog.css') }}" rel="stylesheet" type="text/css" />

<img src="{{ asset('bundles/jamnotasfrontend/images/mentor.png')}}"></img>

Importante, para colocar en su sitio (en el DocumentRoot) los assets, hay que ejecutar el comando de consola:

app/console assets:install target [--symlink]

Variables globales en las plantillas¶

  • app.security - El contexto de seguridad
  • app.user - El usuario actual.
  • app.request - El objeto request.
  • app.session - El objeto sesión.
  • app.environment - El entorno actual (dev, prod, etc).
  • app.debug - True en modo debug.

Sobreescribir las plantillas de los bundle¶

Si quieres cambiar alguna plantilla de un bundle siguiendo buenas prácticas de programación, NO la toques directamente. Eso provocaría un cambio en un código que no es tuyo y cuya evolución, por tanto, no controlas. Si dicho código evoluciona y quieres actualizarlo, perderías en el proceso los cambios que tú has hecho.

Symfony2 permite sobreescribir las plantillas de un bundle sin necesidad de tocarlo directamente. Si, por ejemplo, quieres sobreescribir la hipotética plantilla:

src/Acme/BlogBundle/Resources/views/Blog/index.html.twig

coloca tu versión de la misma en:

app/Resources/AcmeBlogBundle/views/Blog/index.html.twig,

Symfony2, usara esta última en su lugar.

Nota

Recuerda borrar la cache
